Running My Own Model & Harness

Phase 1 was the map; Phase 2 was the parts; Phase 3 is proving it — by running a model and an agent on my own machine, no Claude, no internet, no API key.

1 · A peek under the hood (the gist, not the math)

Attention / transformers

The design inside a modern model. When predicting the next word, it weighs which earlier words matter most — "attention" is that weighing. Stack a lot of it and you get a transformer, the engine behind today's models.

Embeddings

Turning text into a list of numbers that captures meaning, so software can compare and search by meaning (not just exact words). "King" and "queen" land near each other.

Why context costs money & time (KV cache)

As the model reads, it saves its work on earlier tokens (the KV cache) so it doesn't redo it. But the longer the context, the more there is to hold and process — so longer context = more compute = more cost and more lag.

How a model actually runs

An inference engine is the program that runs a model on your hardware — I used Ollama (others: llama.cpp, vLLM). It leans on your GPU/VRAM (on a Mac, the shared memory). Quantization = storing the model's numbers at lower precision so it fits smaller, cheaper hardware — at some cost to quality. My 1-billion-parameter models are quantized, which is why they fit in 8 GB.

How you'd know a model is "good"

Eval = testing it on real tasks; cost / latency = price and speed; observability = being able to see what it did. My bake-off below is a tiny eval.

2 · My local setup

Hardware: Apple M1, 8 GB RAM. Engine: Ollama. Models: llama3.2:1b (1.3 GB) and qwen2.5:1.5b (986 MB).

These run entirely on the laptop — free, offline, nothing leaves the machine. They're noticeably weaker and (sometimes) slower than Claude. That contrast is the lesson.

3 · ★ The swap — model and harness are separate

I took my Phase-2 agent and pointed it at the local model instead of Claude's API: tiny_agent.py (Claude) → tiny_agent_local.py (local).

The only real change is call_model(). The loop, the context, and the calculator tool are identical. Yank out the brain (Claude), drop in a different one (llama3.2), and the agent still runs. That proves it: the harness (the body) and the model (the brain) are separate, swappable parts.

What I noticed: the local model called the calculator once for the multiplication, then did the subtraction in its own head instead of calling the tool again. It got lucky and was right — but "small model does math in its head" is exactly where weaker models slip. A stronger model tends to use the tool for every step.

4 · Bake-off — two local models, same prompts

Run via bakeoff.py. Same prompts, both models:

Promptllama3.2:1bqwen2.5:1.5bRight answer
Bat & ball reasoning trap$0.15 ✗$0.10 ✗$0.05
Apollo 11 year (fact)1969 ✓1969 ✓1969
List exactly 3 fruits (instruction)✓✓—
Finding: both small models nail facts and simple instructions, but both fail the reasoning trap (the right answer is $0.05; they fall for the obvious-but-wrong $0.10-ish trap). Takeaway: tiny local models are fine for recall and formatting, weak on careful reasoning — which is what the big hosted models are still better at.

5 · Honest comparison — my setup vs. Claude Code

PillarMy local setupClaude Code
Modelllama3.2:1b — tiny, weak, but free & privateClaude — huge, strong, hosted (costs $, data leaves machine)
Contextsmall windowvery large window
Harnessmy ~50-line loopfull harness: plan mode, subagents, memory, compaction
Toolsone calculatorfiles, bash, web, MCP servers, hooks, permissions

The trade: local = free, private, offline, but weaker and barer. Hosted = smart, fast, fully-tooled, but costs money and your data leaves your computer. Same architecture in both — just different-sized parts.
