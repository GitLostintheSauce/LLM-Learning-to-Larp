Phase 2 · Inside the pillars

The Four Pillars, Cracked Open

Phase 1 was the four boxes. Phase 2 splits each box into its real parts — explained in my own words, each with a real example from a tool I actually used.

The Model splits into…

The text-predictor itself, in parts.

Tokens
Tokens are the chunks — often pieces of words — that the model actually reads; text gets broken apart into them (e.g. “unbelievable” → a few tokens).
Training vs. inference
Training is when the model learns from a large dataset to infer what comes next, then gets validated on new unseen data. Inference is actually using the finished, frozen model to get an answer.
Temperature & sampling
How probabilistic (random vs. focused) you want the model’s answers to be.
Open vs. closed models
Open models let you download the architecture and run it yourself; closed models run on separate servers (API-only).

The Context splits into…

Everything the model can see, in parts.

Roles
Roles define which context is which and whom it is from (system / user / assistant).
The context window
The context window is how many tokens a model can hold within a given session.
Prompt engineering
How to prompt an AI model, based on context, to receive better results.
Retrieval (RAG)
Retrieval-Augmented Generation is when the harness fetches relevant information from the outside world and adds it to the context so the model can understand the prompt better.

The Harness / Loop splits into…

The program that wraps the model and loops it. I built one by hand: tiny_agent.py.

Think → act → observe loop
Real example: the while True: loop in my agent.
The Think → Act → Observe loop (also known as the ReAct pattern): the model thinks (reasons), acts (asks for a tool), and observes (reads the result), repeating to work through a task autonomously.
Tool-call protocol
Real example: the model returns a tool_use, my code runs it, sends a tool_result.
The tool-call protocol is the method of the model asking for a tool to be used and the harness actually going and using the tool.
Subagents
Subagents are separate agents working at the same time to split up a task.
Memory & context management
Memory & context management is a feature of the harness: it adds context to the model and also gathers/organizes context for the model within the loop.

The Tools & World splits into…

The outside world the harness acts on. I used every part of this in the with/without experiment.

Tool definitions
Real example: the calculate schema in my agent.
Tool definitions are concrete descriptions of tools so that the model can call on the harness to use them.
MCP
Real example: the visualize MCP server that drew my Phase-1 diagram.
MCP (Model Context Protocol) is an open-source standard way that the model connects to the tools / outside world.
Permissions & safety
Real example: the allowlist in .claude/settings.json (kitted run).
Permissions & safety are the guardrails that protect the user from the model/agent doing something the user is not comfortable with; they’re built in and the user can change them.
Hooks
Real example: validate.py running after every edit (kitted run).
Hooks are automatic actions the harness runs before/after a step — plain, deterministic code, not instructions to the model. (Skills are instructions for the model; hooks are automatic actions for the harness.)
