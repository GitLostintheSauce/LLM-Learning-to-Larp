Phase 2 · Inside the pillars

The Four Pillars, Cracked Open

Phase 1 was the four boxes. Phase 2 splits each box into its real parts — explained in my own words. Where I had something slightly off, I've kept my original and added the correction as a marked edit.

The Model splits into…

The text-predictor itself, in parts.

Tokens
How the model breaks apart context.
✏️ Edit: closer — tokens are the chunks (often pieces of words) the model actually reads; text gets broken apart into them (e.g. “unbelievable” → a few tokens).
Training vs. inference
Training is when the model is trained on a large data set to learn/infer what comes next, and then the model gets validated with new unseen data.
✏️ Edit: that's the training half — the contrast is inference: actually using the finished, frozen model to get an answer (what happens every time you chat with it).
Temperature & sampling
How probabilistic you want your model's answers to be.
Open vs. closed models
Open models allow you to download the architecture while closed models are run on separate servers.

The Context splits into…

Everything the model can see, in parts.

Roles
Roles define which context is which and whom it is from.
The context window
Context window is how many tokens a model can hold within a given session.
Prompt engineering
How to prompt an AI model based on context to receive better results.
Retrieval (RAG)
Retrieval-Augmented Generation is when the harness fetches relevent information from the outside world and adds it to the context so that the model can understand the prompt better.

The Harness / Loop splits into…

The program that wraps the model and loops it. I built one by hand: tiny_agent.py.

Think → act → observe loop
Real example: the while True: loop in my agent.
The Think → Act → Observe loop, also known as the ReAct pattern, is a method to autonomously plan and replicate critical thinking.
✏️ Edit: more precisely, the loop is the model thinks (reasons), acts (asks for a tool), and observes (reads the result), repeating to work through a task.
Tool-call protocol
Real example: the model returns a tool_use, my code runs it, sends a tool_result.
The tool-call protocol is the method of the model asking for a tool to be used and the harness actually going and using the tool.
Subagents
Subagents are separate agents working at the same time to split up a task.
Memory & context management
Memory and context management is a feature of the harness in that it adds context to the model and also gathers/organizes context for the model within the loop.

The Tools & World splits into…

The outside world the harness acts on. I used every part of this in the with/without experiment.

Tool definitions
Real example: the calculate schema in my agent.
Tool definitions are concrete descriptions of tools so that the model can call on the harness to use them.
MCP
Real example: the visualize MCP server that drew my Phase-1 diagram.
MCP (Model Context Protocol) is an open-source standard way that the model connects to the tools / outside world.
Permissions & safety
Real example: the allowlist in .claude/settings.json (kitted run).
Permissions and safety are the guard rails that protect the user from the model/agent hallucinating or doing something the user is not comfortable with, and these are built in and the user can change them.
Hooks
Real example: validate.py running after every edit (kitted run).
Hooks are kind of like built-in skills that Claude Code has that give a standard practice of what to tell the agent/model to do before/after every task.
✏️ Edit: not quite — a hook isn't a skill and doesn't instruct the model. It's an automatic action the harness runs before/after a step (deterministic code, e.g. validate.py firing after every edit). Skills = instructions for the model; hooks = automatic actions for the harness.

My hand-built agent

The ★ exercise: tiny_agent.py — a loop around Claude that runs one tool until done.

tiny_agent.py is a small program that is a minimal agent — a loop that wraps the model (Claude) and one calculator tool. Each time through the loop, the harness sends the whole conversation to the model; if the model asks for the calculator, the harness runs it and feeds the result back. The key moment is when the model asks for a tool (returns a tool_use) instead of giving its final answer (end_turn). The model can't run the calculator itself because it only predicts text — it can't actually do anything — so the harness is what executes the tool.

With / without: what the knobs bought me

Same app built twice — naked vs. kitted.

Trying the two builds, they felt about the same — the task was simple and I wasn't after anything specific, so the customizations didn't have a big effect. (The knobs pay off more as the task gets bigger and more specific.)

The knob that surprised me most was the hook, because the same check passed the kitted build but failed the naked one, catching a flaw I had noticed when I tried playing around with the "without" version. (When you do the number order thing, it doesn't click on anything.)

If I kept building this app, the first knob I'd keep is prob CLAUDE.md, because it writes the rules and scoring down once, so everything I build after stays consistent.
